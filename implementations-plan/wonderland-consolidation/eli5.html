wonderland-consolidation

Wonderland (an external team) built three Aztec projects and their contract is ending, so we're adopting the code. This plan moves all three into ONE new repository (ecosystem-tooling) organized your preferred way (bun, biome, per-package CI), fixes the things that would break users when we publish under the new name, and finishes with the actual goal: all three packages published to npm at Aztec v5.0.0-rc.2 under @alejoamiras/… — proving the whole pipeline works end to end.

Why this tier was chosen

mid (you called it; the rubric agrees). Of the six risk dimensions, only external coupling scores clearly HIGH (we depend on Aztec's fast-moving release candidates, npm's publishing rules, and GitHub CI behavior). Blast radius and irreversibility are moderate — the new packages live under a fresh name, so nobody is forced onto them, and release-candidate publishes are low-stakes. One HIGH → mid: one plan, one competing outline, two independent audits, one final fresh-eyes audit.

What the audits changed (short version)

Three independent reviews (Codex, a fresh Claude auditor, then a final fresh-eyes Codex on the consolidated plan) all landed on "approve, with conditions" — and every condition was adopted. The biggest catches: (1) today Wonderland's publish step secretly strips all scripts and dependencies from what goes to npm — if we'd published naively, every consumer's install would have broken on a leftover git-hook script; the plan now has an explicit "manifest curation" checklist. (2) The auditors checked Wonderland's in-flight rc.2 upgrade PRs against live CI: standards is green, fee-payment's benchmark job is actually failing (a renamed function in Aztec rc.2), and benchmark has no test CI at all — so we know exactly which upstream work to trust and which fix we'll write ourselves. (3) The final pass tightened the release itself: the release workflow re-tests the exact code it's about to publish (never trusting an earlier run), the secure publishing route is locked to the same protected environment as the bootstrap token, and "benchmarks ran" now requires actual non-empty comparisons (the tool would otherwise happily report success on nothing).

Phases

Phase 1: Repo bootstrap

Create the empty monorepo with all your house tooling: bun workspaces, biome, commit hooks, supply-chain guardrails (the 7-day "quarantine" for new npm packages, with a carefully-scoped exception for Aztec's own fast-moving packages), plus attribution and licenses crediting Wonderland. Also: verify up front that your npm account can actually publish under @alejoamiras.

Proves itself by: install + lint + workflow-lint all clean, commit hooks demonstrably firing, npm account check passing.

Phase 2: Bring in aztec-benchmark

The smallest project goes first because the other two depend on it (as a library AND as shared CI machinery). Its GitHub benchmark workflows get copied in and adapted — including a security fix: the job that runs untrusted PR code is separated from the job allowed to post comments.

Proves itself by: the package builds, its CLI runs, lint and workflow-lint clean.

Phase 3: Bring in aztec-standards

The contract-standards library (tokens, vaults, NFTs…) moves in, gets renamed, and gets a proper "front door" (explicit entry points) that still preserves every import path existing users rely on. We also run a cheap throwaway CI experiment to answer the one big open infrastructure question early: are GitHub's free runners strong enough to run the Aztec local network?

Proves itself by: 329 Noir tests + 29 integration tests green against a local Aztec network, benchmarks complete, CI experiment timed and recorded.

Phase 4: Bring in aztec-fee-payment

The fee-payment contract + SDK moves in and finally gets ONE canonical name (it currently has three conflicting ones). Two known landmines get defused: a test-config path that breaks inside a monorepo, and hardcoded network addresses that prevent running multiple test environments side by side.

Proves itself by: all Noir + integration tests green (including the L1 bridging tests), plus one test run on NON-default ports to prove the hardcoding is really gone.

Phase 5: CI + first real publish (canary)

Build the full CI system in-house (Wonderland's lived in a separate repo that dies with the fork): per-package test pipelines, a hardened Aztec-installer action, and the release machinery. Then actually publish throwaway "canary" versions to npm — this creates the three package names, proves the pipeline end to end, and unlocks tokenless publishing (npm requires a first publish before the secure OIDC route can be configured). Two small manual steps are yours here (create a temporary npm token; later flip on "trusted publisher" and revoke it).

Proves itself by: all CI green on a real PR; canary versions installable via plain npm in a clean directory, with every legacy import path verified working; dist-tags checked so canaries can never masquerade as real releases.

Phase 6: The rc.2 release 🎯

A new one-command version-bump script updates the ~40 scattered places the Aztec version lives, and prints a supply-chain report for the PR. We apply Wonderland's in-flight rc.2 fixes where their CI proved them, and write the fix they didn't (the failing fee-payment benchmark). Then: publish v5.0.0-rc.2 of all three packages with cryptographic provenance. This is the deliverable.

Proves itself by: full test suite green at rc.2 BEFORE publishing (re-run inside the release workflow on the exact release commit); afterwards all three packages resolvable on npm with the right tags, provenance attestations visible, and a clean-room npm install + import check.

Phase 7: Docs & wrap

README with a migration guide for users of the old packages, CI runbook, roadmap of deliberately-deferred work, and cleanup of dead config carried over in the snapshots.

Proves itself by: full lint + test suite one last time; docs match reality.

Human context

Three things worth understanding: (1) These packages version-track Aztec itself — package v5.0.0-rc.2 means "works with Aztec 5.0.0-rc.2". The monorepo makes that lockstep structural instead of manual, which is exactly the failure currently biting Wonderland's own fee-payment PR (their benchmark tool is one version behind its host repo). (2) Publishing under a new scope is a fresh start with obligations: nothing breaks for old users (old packages stay frozen), but they'll never find the new ones unless Wonderland posts a deprecation notice while they still can — that's a one-message favor to ask now. (3) npm's most secure publishing mode can't create a package's first version, so there's a deliberate one-time-token bootstrap, then the token is destroyed.
