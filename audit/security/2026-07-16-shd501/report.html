The contract core is clean — the important result

The Noir smart-contract logic that actually moves money — how fees get charged, how a bridged deposit is proven and can't be spent twice, how the cryptography ties a deposit to its rightful owner — was examined independently by two different AI model families, and both found nothing. Both even re-derived the contract's cryptographic "domain separator" by hand and got the same value it uses. For a contract with no admin, no upgrade button, and no undo, that convergent all-clear on the money-handling core is the headline.

None of the four findings is Critical or High, and none needs an emergency patch. The published contract is not broken. Everything below is about the surrounding TypeScript toolkit and how the package is bundled.

What to actually do (all deferrable to the next release)

F-001 (the one real issue): give SDK users a way to cap how much a transaction can cost, so a hostile network node can't inflate their fee. F-002: make the "correct address to send funds to" check enforce itself at runtime, not just in the test suite. F-003 + F-004: delete or stop shipping unused/test-only code that rides along in the published package. All four are small, additive, and safe to bundle into the next scheduled release — no hotfix warranted.

Findings

Medium F-001 A hostile RPC node can silently drain a user's fee balance
Found by Claude + Codex confidence: high CWE-20 / CWE-345 fix: hours · prod-reachable

What it is. Before sending a transaction, apps call a helper (estimateGasSettings) that asks the connected Aztec network node "how much does gas cost right now?" and trusts the answer with no upper limit. Separately, this fee contract charges the full estimated maximum and never refunds the difference (that's intentional — private contracts can't refund). Put together: whatever the node says the fee is, that's what gets permanently taken from the user.

What an attacker would do. Run (or compromise, or sit in the middle of) the network node a wallet talks to — a well-worn threat in crypto, where users routinely point at shared public RPC endpoints. Return a gas price 100× the real one. The user's app faithfully turns that into the transaction's fee cap, signs it, sends it, and the contract burns that inflated amount from the user's internal balance — every transaction, until their balance is gone. It hits one user at a time, but every user of a dApp pointed at that bad node is affected the same way.

The fix. Add an optional "I won't pay more than X" ceiling to the estimate helper, checked on the user's own machine before the number is trusted — independent of anything the node claims. It's a small, backwards-compatible addition to the SDK; ship it in the next release. The contract itself needs no change.

Technical trace

Trace: estimateGasSettings (root export, src/ts/index.ts:42) reads getCurrentMinFees() and getNodeInfo().txsLimits.gas from a caller-supplied node, applies only a fixed 1.2× multiplier + finite/≥0 padding validation, and returns them as the tx's declared GasSettings — no absolute ceiling (gas.ts:189,192-195,183-234; the padAndClampGas clamp is sourced from the same node, so it is not independent). On-chain, get_max_gas_cost recomputes from the declared settings (fpc_lib/src/lib.nr:17-27) and pay_fee deducts the full amount with an explicit "Does not refund unused gas" (main.nr:46-58).

Fix: add an optional maxAcceptableGasCost (or max-fees-per-gas cap) parameter to estimateGasSettings, throwing client-side if the node-derived cost exceeds it. Blast radius is per-signing-user (GasSettings is per-tx), not a cross-user or FPC-pool drain. Both models traced this independently and converged; main agent re-verified gas.ts:183-234 and index.ts. Defer to next lockstep release (additive). Effort: hours.

Low F-002 The "send funds here" address guard only checks itself
Found by Claude (Codex: same facts, rated test-gap) confidence: moderate CWE-345 fix: hours

What it is. This contract lives at one fixed address that users bridge real funds to; sending to a stale or wrong address means the money is unrecoverable. The safety check that's supposed to prevent that only compares the address against other copies of itself in the same repo (a JSON file, the README, the package version). Nothing anchors it to an outside source of truth, and the actual function apps call to register the contract accepts any address with no check at all.

What a user would trigger. Not a remote attacker — but a single coordinated edit (or a compromised publish pipeline) could change the JSON and README together, pass every test green, and quietly point every future user at a different address. Or an integrator copies a stale address out of old docs; nothing stops them, and because this contract has no on-chain "does this exist?" check, the mistake surfaces only after funds are lost.

The fix. Let apps import the canonical address as real code (not copy it from prose), and add an opt-in "assert this matches canonical" check to the register function so a wrong address throws instead of silently succeeding. Small SDK addition; next release.

Technical trace

Trace: canonical.test.ts:22-47 asserts only version==pin, address==derivation, README-line-embeds-address (all same-repo inputs, no external anchor) and never executes registerPrivateContract. registerPrivateContract (deploy.ts:23-28) takes an arbitrary salt with no canonical cross-check. canonical-deployment.json is not in the exports map (package.json:13-31), so integrators hand-copy the value from README prose. Codex independently confirmed the same test-coverage gap and the documented "noncanonical salt → valid-but-different address" behavior, rating it a gap rather than a finding.

Fix: export the canonical record as a subpath; add opt-in assertCanonical to registerPrivateContract; optionally anchor the test against the prior published tarball's record. Defer. Effort: hours.

Low F-003 Unused artifact-registry code ships with weak safety checks
Found by Claude + Codex confidence: high (not reachable) CWE-494 / CWE-918 fix: hours

What it is. The package contains a module for fetching contract definitions from an external web service. The fetch path trusts whatever the server returns with no validation, and the upload path talks to a URL from an environment variable with no authentication and following redirects. The good news: the fetch path has zero callers anywhere — it's dead code — and the upload path is only used by a manual developer command, never in automation.

What an attacker would do. Nothing today — there's no way to reach the dangerous path through the package's normal interface. The concern is forward-looking: if a future change wires this in without noticing the missing validation, a malicious registry could feed a poisoned contract definition and misdirect funds. Shipping the typings also makes it easy for someone to auto-import it and build on it, inheriting the gap.

The fix. Simplest and safest: delete the module, since nothing uses it. If it's kept, add proper validation (schema + hash) to the fetch, force HTTPS with no auto-redirects on the upload, and stop shipping it in the package.

Technical trace

Trace: artifactRegistry.ts:131-169 — return artifact as NoirCompiledContract, bare cast of a network response, no schema/hash/signature check, registry-first over local file. artifactRegistry.ts:27-53 — env-controlled base URL (no scheme allowlist), upload POST with no auth, default redirect-follow. Both models grep-confirmed fetchArtifactFromRegistry/loadArtifactWithRegistryFallback have zero callers repo-wide; the only caller of anything here is the manual scripts/upload-artifact.ts dev CLI (not in any workflow). Ships in dist but not in exports (Node blocks the normal import with ERR_PACKAGE_PATH_NOT_EXPORTED). AZTEC_ARTIFACT_REGISTRY_STRICT governs upload error-swallowing only, not fetch validation.

Fix: delete if unused (removes the forward-risk); else add zod + classId-hash validation, https:-only allowlist, redirect:'manual', and exclude from the tarball. Defer. Effort: hours.

Low F-004 A test-only contract (1.2MB) ships to every consumer
Found by Claude + Codex confidence: high CWE-749 fix: minutes–hours

What it is. The published package bundles a "Counter" contract that exists purely for internal benchmarks and tests — its 1.2MB compiled file plus a fully-usable deploy binding — even though it's not part of the package's advertised interface. Confirmed by packing the real tarball.

What a user would trigger. No direct exploit. But a dependency-scanning tool or a curious developer finds a deployable "Counter" contract inside a package whose job is fee payment, assumes it's a supported, audited part of the product, and builds on benchmark-grade code that was never hardened for production. Plus it needlessly bloats every install.

The fix. Curate what the package ships so only the real fee-payment contract and the public toolkit go out — drop the test contract and the unused registry module from the tarball. Quick config change.

Technical trace

Trace: package.json:34-39 whitelists all of target and dist wholesale (only *.json.bak excluded). npm pack --dry-run --json confirmed the 5.0.1 tarball ships target/counter_contract-Counter.json (1,209,208 bytes, test-only) and target/private_contract-PrivateFPC.json (2.4MB); ~3.6MB unpacked before dist. A deployable CounterContract binding (src/artifacts/Counter.ts) and artifactRegistry.js/.d.ts also ship, none in the exports map.

Fix: curate files/add .npmignore to exclude the Counter artifact + binding and artifactRegistry.*; ship only the PrivateFPC artifact + exported SDK. Defer; bundle with F-003. Effort: minutes–hours.

Cross-cutting observations

The three money-moving contract functions have no active unit tests

The contract's most security-critical functions — pay_fee, mint, mint_and_pay_fee — have their fast Noir-level unit tests entirely commented out, because the test environment can't simulate a real bridge deposit, runs with zero gas, and can't handle the phase-transition these functions use. They're covered only by slower integration tests that need a full live network running, and one of them (mint_and_pay_fee) only by a benchmark that doesn't assert correctness. This isn't a vulnerability, but the quickest safety net for the riskiest code is switched off — a mock-oracle test harness that lets these run as unit tests would materially strengthen regression protection. (Matches the roadmap's existing "structural test debt" note.)

The math is solid; the exposure is trusting the network node

Every finding of substance lives in the TypeScript trust boundary, not the zero-knowledge circuit. The recurring theme (F-001 especially) is that the SDK should treat the RPC node it talks to as untrusted and give integrators the controls to enforce that. The cryptographic core held up to two independent model families re-deriving it.
