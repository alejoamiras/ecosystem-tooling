## Result: 0 findings

No exploitable CIA/authorization/nullifier-integrity violation with a complete source-to-sink trace was found in this cluster. The design closely follows vetted aztec-nr patterns (the `_subtract_balance`/`recurse_subtract_balance_internal` split mirrors the standard Token-contract recursive-subtraction pattern; the nullifier double-spend guard mirrors the documented FeeJuice-claim-proof pattern). Below is what was specifically checked and ruled out, plus one investigation that looked like a real bug initially and was disproven after tracing the actual protocol fee-charging code — documented so it isn't re-litigated by another pass.

### Checked and clear

1. **Phase safety (`pay_fee` / `mint_and_pay_fee`).** Both are single, straight-line, branch-free (w.r.t. the fee-payer sequencing) Noir circuits: balance deduction/credit always executes *before* `set_as_fee_payer()`/`end_setup()` in program order, and Noir/ACIR circuits are all-or-nothing — a failed assert anywhere invalidates the entire witness, so there is no state where "fee payer set but balance not deducted" (or vice versa) can be reached; a partially-executed proof does not exist. `set_as_fee_payer()` itself asserts `!in_revertible_phase()` (`private_context.nr:548`), which fails closed if some other function in the tx already called `end_setup()` — this can only make the caller's own tx unprovable, not corrupt state for anyone else.
2. **Nullifier double-spend.** `feejuice_nullifier` (main.nr:80-88, 140-148) is deterministically bound to `(fpc_address=self.address, amount, salt, claimer=self.msg_sender(), leaf_index, chain_id, version)`. Both `mint` and `mint_and_pay_fee` compute and push the *identical* raw nullifier for the same tuple, and `push_nullifier_unsafe` (private_context.nr:395-398) is auto-siloed by the kernel with the calling contract's address — this makes the FPC's nullifier tree entry (siloed with the FPC address) provably distinct from FeeJuice's own tree entry (siloed with `FEE_JUICE_ADDRESS`, read via `compute_nullifier_existence_request(feejuice_nullifier, FEE_JUICE_ADDRESS)` in `nullifier/utils.nr`), so there's no collision between the "proof of bridge claim" nullifier and the "prevent double-mint" nullifier. A second `mint`/`mint_and_pay_fee` call (or a second call within the same tx) with the same arguments recomputes the same FPC-siloed nullifier and is rejected by the standard duplicate-nullifier check (same-tx: kernel; cross-tx: base rollup non-membership) — this is enforced framework behavior, not bypassed anywhere in this contract.
3. **Claimer binding.** `claimer` in both `mint` and `mint_and_pay_fee` is hard-bound to `self.msg_sender()` — it is never an attacker-suppliable parameter for "credit account X". Since `derive_bridge_secret(salt, claimer)` folds `claimer` into the nullifier, an attacker cannot claim someone else's real L1 deposit: their own `feejuice_nullifier` computation always uses *their own* address, and would only coincidentally match a real depositor's already-emitted nullifier via a poseidon2/sha256 preimage collision (astronomically infeasible; deferred to the crypto-derivation cluster, but structurally the binding holds).
4. **Authorization surface.** Every state-mutating function in this contract (`pay_fee`, `mint`, `mint_and_pay_fee`, `_deduct_max_gas_cost`, `_subtract_balance`) resolves its `account`/`claimer` parameter from `self.msg_sender()` at the entrypoint, never from a free-standing caller-supplied address for a third party. `_deduct_max_gas_cost`/`_subtract_balance` are `#[internal("private")]` (not part of the external ABI — callable only via in-circuit calls from this contract's own private functions per `macros/functions/mod.nr:372-392`); `recurse_subtract_balance_internal` is `#[external("private")] #[only_self]`, externally reachable in the ABI but restricted to `msg_sender == self.address()`, which only the FPC's own call graph can satisfy. No cross-account balance mutation path exists.
5. **`_subtract_balance` accounting (main.nr:191-202).** `try_sub` (balance_set.nr:78-98, unmodified aztec-nr library code) returns `subtracted`, which is either `>= amount` (change = `subtracted - amount`, no underflow) or `< amount`, in which case `assert(subtracted > 0)` fails closed on an empty/insufficient balance (no infinite/zero-progress recursion), and the recursive leg subtracts exactly `amount - subtracted` (well-formed, no underflow). `u128` subtraction generally: underflow in Noir/ACIR range-checked arithmetic makes the circuit unsatisfiable rather than silently wrapping, so the explicit `assert(amount >= max_gas_cost, ...)` in `mint_and_pay_fee` (main.nr:101) is a clearer-error redundant guard, not the only thing standing between the code and a wraparound.
6. **`get_max_gas_cost` / gas-settings griefing.** `context.gas_settings()` reads the single canonical `tx_context.gas_settings`, identical across every call in the tx (protocol invariant); the caller who sets it is the same party whose own internal balance gets debited (`sender = self.msg_sender()` in `pay_fee`), so inflating gas limits only harms the caller's own transaction/balance, not another user's. Investigated in depth whether excluding `teardown_gas_limits` from the formula under-charges relative to what the protocol actually bills the fee payer (see below) — traced fully and ruled out.
7. **`balance_of` (main.nr:217-220).** Unconstrained utility function, not referenced anywhere else in `main.nr`'s constrained logic — no misuse of an unconstrained return value in a constrained code path was found in this cluster.
8. **`MessageDelivery::onchain_unconstrained()` note delivery (main.nr:102-104, 164-166, 178-180).** Per `aztec-nr/aztec/src/messages/delivery/builder.nr:14-17`, this mode is appropriate "where the sender is incentivized to deliver the message correctly." In every call site here, sender and recipient of the note are the same party (`self.msg_sender()`/`claimer`/`account`), so there is no trust gap — the note owner is the one constructing/proving their own delivery, and failing to do so only hurts themselves. Not a confidentiality/redirection vector.
9. **Reentrancy via `call_self`/recursion (main.nr:200, 208-212).** Aztec private execution has no mutable global storage observable mid-transaction the way EVM reentrancy works; the recursive `_subtract_balance` → `recurse_subtract_balance_internal` → `_subtract_balance` chain only ever operates on the same `account` bound at the entrypoint (see #4), so there is no cross-account reentrancy surface. (Unbounded recursion depth on a single, self-fragmented balance is a possible self-DoS for that same account owner, but requires no adversary and cannot be triggered against a victim — nothing in this contract lets one account credit notes into another account's `BalanceSet`, so a dust-bombing griefing vector doesn't exist either.)

### Investigated and ruled out: teardown-gas exclusion in `get_max_gas_cost`

`fpc_lib/src/lib.nr:17-27` computes `max_fee_per_da_gas * da_gas_limit + max_fee_per_l2_gas * l2_gas_limit` from `gas_settings.gas_limits` only, explicitly excluding `gas_settings.teardown_gas_limits` (comment: "the kernel's gas meter already includes teardown allocation in the overall gas limit ... so adding teardown_gas_limits again would double-count it"). This is also documented as a deliberate 2026-03-04 fix in `docs/private-product-requirements.md` (v1.3: "Teardown double-counting fix") and mirrored in the TS SDK (`src/ts/utils/gas.ts:137-140`, `maxGasCostFor`).

This looked like a real under-charging bug on first read: `noir-protocol-circuits/.../gas_meter.nr:90-94` shows `teardown_gas_limits` is a genuinely separate, additive field in `GasSettings` (not folded into `gas_limits`), and for private-to-public transactions (`is_for_public=true`) with a non-empty `public_teardown_call_request`, `meter_gas_used` adds the *full* `teardown_gas_limits` on top of the metered non-teardown usage. If the FPC only charges `gas_limits.compute_fee(...)`, that looks like it omits the teardown component entirely.

Full trace disproves this: `private-kernel-lib/.../tail_to_public_output_validator.nr:162-167` asserts `gas_used.within(gas_settings.gas_limits)` where `gas_used` is that same teardown-inclusive metered value — i.e. the protocol *requires* `gas_limits` to already be large enough to cover (non-teardown usage + teardown_gas_limits) or the transaction fails to validate at all. Combined with `compute_effective_gas_fees` (`rollup-lib/.../fees.nr:21-41`) always producing `effective_fees <= max_fees_per_gas` per gas unit, and `compute_transaction_fee = gas_used.compute_fee(effective_fees)` (`fees.nr:43-50`, consumed via `private_to_rollup.gas_used` in `private_tx_effect_builder.nr:49-53` for the pure-private path), the dominance `gas_used <= gas_limits` (component-wise) and `effective_fees <= max_fees_per_gas` (per-unit) together guarantee `gas_used.compute_fee(effective_fees) <= gas_limits.compute_fee(max_fees_per_gas)` always — i.e. `get_max_gas_cost()` is provably an upper bound on the real fee, teardown included, in every case where the transaction validates at all. There is no way to construct a validating transaction where the real bill exceeds what `get_max_gas_cost()` collects. The "no refund" design means the FPC is conservative (may over-collect when there's slack between declared limits and actual usage), never under-collects. No finding.

(Note: the public/AVM-path fee for private-to-public transactions is ultimately reported via `avm.transaction_fee` in `rollup-lib/.../public_tx_effect_builder.nr:114`, computed inside AVM/public execution rather than `private_tx_effect_builder.nr`. I did not trace the AVM circuit itself — out of scope for this Noir-entrypoints cluster — but the kernel-side `gas_used.within(gas_limits)` invariant established before handoff to public execution is what the FPC actually relies on, and that invariant is confirmed above at the private-kernel boundary.)

## Not flagged (explicitly out of scope per assignment / no exploit path)

Structural binding of `amount`/`claimer`/`fpc_address`/`chain_id`/`version` into the nullifier was verified as described above.

- Self-inflicted recursion-depth exhaustion in `_subtract_balance` from a fragmented own balance — availability concern against oneself only, no adversary can trigger it against a victim (accounts can't be credited notes by anyone but themselves).
